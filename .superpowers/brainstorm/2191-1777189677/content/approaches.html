3 Hướng kiến trúc cho Recipe & Food Transform System

Dựa trên: Hybrid pipeline · Station Slot mechanic · 1 player (multiplayer-ready). Chọn hướng bạn muốn đi sâu.

A

Flat ScriptableObject List

Ý tưởng: Mỗi công thức là 1 RecipeSO chứa danh sách input + output. Station tự loop qua danh sách để tìm match.

Ưu: Cực đơn giản, dễ hiểu, ít class.
Nhược: Mỗi lần station check phải loop toàn bộ recipe list. Khó scale khi nhiều recipe. Không tái sử dụng được "bước chặt" hay "bước nấu" riêng lẻ — mỗi combo là 1 SO riêng.

⚠ Không recommend cho game có 5 map, nhiều món

B

Food State Graph (Transformation Chain)

Ý tưởng: Food là một state machine. Mỗi FoodStateSO là 1 node, TransformationSO là edge (input state → action → output state). Recipe chỉ là tập hợp final states cần đạt.

Ưu: Cực linh hoạt. Thêm món mới = thêm node + edge. Tái sử dụng transformation (chặt onion dùng chung cho mọi recipe có onion).
Nhược: Abstract hơn, cần hiểu graph traversal. Hơi overkill nếu game không quá phức tạp.

⚡ Mạnh nhất về lý thuyết, nhưng steep learning curve

C

Layered SO + RecipeDatabase ⭐ Recommended

Ý tưởng: Tách rõ 3 layer:
1. Data SOs — FoodStateSO (trạng thái food) + TransformationSO (A + station → B) + RecipeSO (set inputs → meal)
2. RecipeDatabase — ScriptableObject trung tâm, index tất cả recipe/transformation, expose method TryGetTransformation() và TryMatchRecipe()
3. Runtime MonoBehaviours — FoodItem (track current state), StationBase + subclasses (query Database khi slot filled)

Ưu: Cân bằng giữa đơn giản và linh hoạt. Data hoàn toàn tách khỏi logic. Dễ test từng phần. Multiplayer-ready vì logic không phụ thuộc input.
Nhược: Cần thiết kế upfront nhiều class hơn A.

✅ Phù hợp sinh viên học architecture, scale tốt cho 5 map
